Methods In Go

Go has some object-orient programming elements. Method is one of these elements. This article will explain method related concepts in Go.

Method Declarations

In Go, we can (explicitly) declare a method for type T and *T, where T must satisfiy 4 conditions:
  1. T must be defined type;
  2. T must be defined in the same package as the method declaration;
  3. T must not be a pointer type;
  4. T must not be an interface type.

T is called the receiver base type of any method declared for both type T and *T. (Below will explain what are receivers.)

We can also declare methods for alias types of the T *T types specified above, the effect is the same as declaring methods for the T *T types themselves.

If a method is declared for a type, we can say the type has (own) the method.

From the above listed conditions, we will get the conclusion that we can never (explicitly) define method for
  • built-in basic types, such as int and string, for we can't define methods in the builtin standard package.
  • interface types. But an interface type can own methods. Please read Interfaces In Go for details.
  • unnamed types except the pointer types as *T described above, including unnamed array, map, slice, function, channel and struct types. However, if an unnamed struct type embeds other types which have methods, then compiler will implicitly define some methods for the unnamed struct type and the unnamed pointer type whose base type is the unnamed struct type. Please read Type Embedding for details.

A method declaration is similiar to a function declaration, but it has an extra parameter declaration part. The extra parameter is called a receiver parameter. Each method must be declared with one and only one receiver parameter. The receiver parameter must be enclosed in a () and declared between the func keyword and the method name. The type of the receiver parameter is called the receiver type of that method.

Here are some method declaration examples: 
// Age and int are two distinct types.
// We can't define methods for int, but can for Age.
type Age int
func (age Age) LargerThan(a Age) bool {
	return age > a
}
func (age *Age) Increase() {
	*age++
}

// Receiver of defined function type.
type FilterFunc func(in int) bool
func (ff FilterFunc) Filte(in int) bool {
	return ff(in)
}

// Receiver of defined map type.
type StringSet map[string]struct{}
func (ss StringSet) Has(key string) bool {
	_, present := ss[key]
	return present
}
func (ss StringSet) Add(key string) {
	ss[key] = struct{}{}
}
func (ss StringSet) Remove(key string) {
	delete(ss, key)
}

// Receiver of defined struct type.
type Book struct {
	pages int
}
func (b Book) Pages() int {
	return b.pages
}
func (b *Book) SetPages(pages int) {
	b.pages = pages 
}

From the above example, we know that the receiver base types not only can be struct types, but also can be other kinds of types, such as basic types and container types, as long as the receiver base types satisfy the 4 conditions listed above.

In some other programming languages, the receiver parameter name is always either implicit this (C++ and Java) or explicit self (Python) . The two receiver parameter names are not recommended in Go programming.

The receiver of type *T are called pointer receiver, non-pointer reveivers are called value receivers. Personally, I don't recommand to view the terminology pointer as an oppoiste of the terminology value, for pointer values are just special values. But, I am not against using the pointer receiver and value receiver terminologies here. The reason will be explained below.

(Method names can be the blank identifier _. A type can have multiple methods with the blank identifier as names. But such methods can never be called.)

Each Method Corresponds To An Implicit Function

For each method declaration, compiler will declare a corresponding implicit function for it. By the example of the last two methods whose receiver base types are Book listed in the last section, two following functions are implicitly declared by compiler: 
func Book.Pages(b Book) int {
	return b.Pages() // call the corresponding method
}

func (*Book).SetPages(b *Book, pages int) {
	b.SetPages(pages) // call the corresponding method
}

In each of the two implicit function declarations, the receiver parameter is removed from its correponding method declaration and insert the receiver parameter into the normal parameter list as the first one.

The implict method names, Book.Pages and (*Book).SetPages, are both of the form TypeLiteral.MethodName. As identifiers in Go can't contain the period special characters, the two implict function names are not legal identifiers, so the two function can't be dclared explicitly. They can only be declared by compiler implicitly, and they really can be called: 
package main

import "fmt"

type Book struct {
	pages int
}
func (b Book) Pages() int {
	return b.pages
}
func (b *Book) SetPages(pages int) {
	b.pages = pages
}

func main() {
	var book Book
	// Call the two implicit decalred functions.
	(*Book).SetPages(&book, 123)
	fmt.Println(Book.Pages(book)) // 123
}

Implicit Methods With Pointer Reveivers

For each method declared for value receiver type T, a corresponding method with the same name will be implictly declared by compiler for type *T. By the example above, the Pages method is declared for type Book, so compiler will implicitly declare a method with the same name Pages for type *Book. The same name method only contain one line of code, which is a call to the implicit function Book.Pages: 
func (b *Book) Pages() int {
	return (*b).Pages() // call the value method
}

This is why I don't reject to use the value receiver terminology (as the opposite of pointer receiver). After all, when we expliclty declare a method for a non-pointer type, in fact two methods are declared, the explicit one for the non-pointer type, the other implicit one is for the corresponding pointer type.

As the last section has mentioned, for each of the implicitly decalred methods, compiler will also declare a correponding implicit function for it: 
func (*Book).Pages(b *Book) int {
	return b.Pages() // call the pointer method
}

In other words, for each explicitly declared method with a value receiver, two implicit functions and one implicit method will also be declared correspondingly.

Method Signatures And Method Sets

Each declared method has a signature. The signature of a declared method doesn't include the func keyword and the receiver part in the method declaration. Following are the method signatures of the above declared methods: 
Pages() int
SetPages(pages int)

Each type has a method set. The method set of a type is composed of all the signatures of the methods declared, either explicitly or implicitly, for the type, except the ones whose names are the blank identifier _. From the last serction, we know that the method set of a type T must be a subset of the method set of type *T if *T can have methods. Method sets play an important role in the polymorphism feature of Go. About polymorphism, please read Interfaces In Go for details.

The method set of the following types are always blank:
  • built-in basic types.
  • defined pointer types.
  • unnamed pointer types whose base types are interface or pointer types.
  • unnamed array, slice, map, function and channel types.

Method Values And Method Calls

Methods are special functions in fact. When a method is declared for a type, then each value of the type will own an immutable member of an function type. The member name is the same as the method name. The function type is the same as the function declared with the form of the method declaration but without the reveiver part. The member is often called called a member function. The member function can also be called the method of its corresponding value.

A method call is just a call to such a member function. For a value v, its method m can be accessed with the selector v.m.

(For the just mentioned reason, if the base type of the receiver type of a method is a stuct type, then the method can't share the same name with any explicitly declared field of the struct type, except the name is the blank identifier.)

Example of method calls: 
package main

import "fmt"

type Book struct {
	pages int
}

func (b Book) Pages() int {
	return b.pages
}

func (b *Book) SetPages(pages int) {
	b.pages = pages
}

func main() {
	var book Book
	
	fmt.Printf("%T \n", book.Pages)       // func() int
	fmt.Printf("%T \n", (&book).SetPages) // func(int)
	// &book has an implicit method.
	fmt.Printf("%T \n", (&book).Pages)    // func() int
	
	(&book).SetPages(123)
	fmt.Println(book.Pages()) // 123
	// Call the impliict method of &book.
	fmt.Println((&book).Pages()) // 123
}

In the above example, the value book is call the base value of both the receiver arguments of the Pages and SetPages methods.

(Different from C language, there is not the -> operator in Go to call methods with pointer receivers, so (&book)->SetPages(123) is illegal in Go.)

Method calls make Go programs clean and more readable. In fact, the above example can be cleaner. The line (&book).SetPages(123) in the above example can be simplified to book.SetPages(123). But how can this happen? After all, the value book has not a method called SetPages. Aha, this is just a syntax sugar to make progrmaming convenient. This sugar only works for addressable values of type Book. Compiler will automitically take the address of the addressable values when they are passed as the receiver arguments of the SetPages method. 
 ...

// Function results are not addressable in Go.
func MakeBook() Book {
	return Book{}
}

func main() {
	var book Book
	book.SetPages(123) // <=> (&book).SetPages(123)
	fmt.Println(book.Pages()) // 123

	// error: MakeBook() is not addressable.
	MakeBook().SetPages(123)
}

As above just mentioned, when a method is declared for a type, each value of the type will own a member function. Zero values are not exceptions, whether or not the zero values can be represented by nil.

Example: 
package main

type StringSet map[string]struct{}
func (ss StringSet) Has(key string) bool {
	_, present := ss[key] // Never panic here,
	                      // even if ss is nil.
	return present
}

type Age int
func (age *Age) IsNil() bool {
	return age == nil
}
func (age *Age) Increase() {
	*age++ // If age is a nil pointer, then
	       // dereferencing it will panic.
}

func main() {
	_ = (StringSet(nil)).Has   // will not panic
	_ = ((*Age)(nil)).IsNil    // will not panic
	_ = ((*Age)(nil)).Increase // will not panic
	
	_ = (StringSet(nil)).Has("key") // will not panic
	_ = ((*Age)(nil)).IsNil()       // will not panic

	// This line will panic. But the panic is not caused
	// by invoking the method. It is caused by the nil
	// pointer dereference in the method body.
	((*Age)(nil)).Increase()
}

Receiver Parameters Are Copies Of Receiver Arguments

Same as general parameters, the receiver parameter within a method body is just a copy of the receiver argument passed by the outside caller.

For the receiver parameter within a method body is just a copy of the receiver argument passed by the outside caller, the the direct part of outside receiver argument can't be modified in the method body through modifying the receiver parameter, whether or not the receiver is a pointer one or a value one. But the modifications on the value referenced by a pointer receiver (or the underlying values referenced by the direct part of the receiver) can be reflected to the outside of the method body.

An example: 
package main

import "fmt"

type Book struct {
	pages int
}

func (b Book) SetPages(pages int) {
	b.pages = pages
}

func main () {
	var b Book
	b.SetPages(123)
	fmt.Println(b.pages) // 0
}
Another example: 
package main

import "fmt"

type Book struct {
	pages int
}

type Books []Book

func (books Books) Modify() {
	// Modifications on the underlying part of the receiver
	// will be reflected to outside of the method.
	books[0].pages = 500
	// Modifications on the direct part of the receiver
	// will not be reflected to outside of the method.
	books = append(books, Book{789})
}

func main () {
	var books = Books{{123}, {456}}
	books.Modify()
	fmt.Println(books) // [{500} {456}]
}
Sone off topic, if the two lines in the orders of the above Modify method are exchanged, then both of the modifications will not be reflected to outside of the method body. 
func (books Books) Modify() {
	books = append(books, Book{789})
	books[0].pages = 500
}

func main () {
	var books = Books{{123}, {456}}
	books.Modify()
	fmt.Println(books) // [{123} {456}]
}

The reason here is that the append call will allocate a new space to storage the elements of the slice receiver.

To make both of the modifications be reflected to outside of the method body, the receiver of the method must be a pointer one. 
func (books *Books) Modify() {
	*books = append(*books, Book{789})
	(*books)[0].pages = 500
}

func main () {
	var books = Books{{123}, {456}}
	books.Modify()
	fmt.Println(books) // [{500} {456} {789}]
}

Declare A Methods With Pointer Receiver Or Value Reveiver?

Firstly, from the last section, we know that sometimes we must define methods with pointer receivers.

In fact, we can always define methods with pointer recievers without any logic problems. It is just a matter of program performance that sometimes if is better to define methods with value receivers.

For the cases value receivers and pointer receivers are both acceptable, here are some factors needed to be considered to make decisions:
  • too many pointer copies may cause heavier workload for garbage colllector.
  • pointer value assignments during garbage collecting will be some costlier than non-pointer value assignments.
  • if the value size of a receiver type is large, then the receiver argument copy cost may be not neglectable. In particular if the passed argument is an interface value (please read polymorphism for details), there will be two copies made in the argument passing. Here we just need to know that pointer values are all small sized values. In fact, for the official Go compiler and runtime, types other than array and struct types are all small sized types. Struct types with very few fields are also small sized.)
  • mixing value receivers and pointer receivers for the same base type, is more likely to cause data races if the defined methods are called across multiple goroutines.
  • values of the types in the sync standard package should not be copied, so defining methods with value receivers for sturct types which embedding the types in the sync standard package is problematic.

Sometimes, it may be hard to make a decision a method should use a pointer receiver or a value receiver, then just choose pointer receiver.

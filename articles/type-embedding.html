Type Embedding

From this article, we know that a struct type can have many fields. Each of the fields is composed of one field name and one field type. In fact, sometimes, a struct field can be composed of one field type only. Such fields are called anonymous fields in Go. An anonymous field is also called a type embedding.

This article will explain the intention of type embedding and all kinds of details in type embedding.

What Does Type Embedding Look Like?

Here is an example of type embedding: 
package main

func main() {
	type S = []int
	var x struct {
		string // a defined non-pointer type
		error  // a defined interface type
		*int   // an unnamed pointer type
		S      // an alias of an unnamed pointer type
		
	}
	x.string = "Go"
	x.error = nil
	x.int = new(int)
	x.S = []int{}
}

In the above example, four types are embedded in the struct type. Each type embedding forms an anonymous field.

In fact, each anonymous field has an implicit name. The unqualified type names of the four anonymous fields are their names. For example, the names of the four anonymous fields in the above examples are string, intm error, and S, respectively.

Which Types Can be Embedded?

The following types can't be embedded:
  • defined pointer types.
  • unnamed types, except unnamed one-level pointer types whose base types are non-interface and non-poiner defined types.

Other types can be embedded.

Following lists some types can and can't be embedded: 
type Encoder interface {Encode([]byte) []byte}
type Person struct {name string; age int}
type Alias = struct {name string; age int}
type AliasPtr = *struct {name string; age int}
type IntPtr *int

// These types can be embedded in a struct type.
Encoder
Person
*Person
Alias
*Alias
AliasPtr
int
*int

// These types can't be embedded.
*Encoder         // base type is an interface type
*AliasPtr        // base type is a pointer type
IntPtr           // defined pointer type
*IntPtr          // base type is a pointer type
struct {age int} // unnamed type
map[string]int   // unnamed type
[]int64          // unnamed type
func()           // unnamed type

No two fields are allowed to have the same name in a struct, there are not exceptions for annonymous struct fields. By the anonymous field implicitly naming rules, an unnamed pointer type can't be embedded along with its base type in the same struct type. For example, int and *int can't be embedded in a struct type at the same time.

Generally, it is only meaningful to embed types who have fields or methods. The following sections will explain why.

What Is The Meaningfulness Of Type Embedding?

The main purpose of type embedding is to extend the functionalities of the embedded types into the embedding type, so that we don't need to reimplement the functionalities of the embeded types for the embedding type.

Many other object-oriented programming languages use inheritance to achieve the same goal of type embedding. Both mechanisms have their own benefits and drawbacks. Here, this article will not discuss which one is better. We should just know Go chose the type embedding mechanism, and there is a big difference between the two:
  • If a type T inherits another type, then type T obtains the abilities of the other type. At the same time, each value of type T can also be viewed as a value of the other type.
  • If a type T embeds another type, then type other type becomes a part of type T, and type T obtains the abilities of the other type, but none values of type T can be viewed as values of the other type.
Here is an example to show how an embedding type extends the functionalities of the embedded types. 
package main

import "fmt"

type Person struct {
	Name string
	Age  int
}
func (p Person) PrintName() {
	fmt.Println("Name:", p.Name)
}
func (p *Person) SetAge(age int) {
	p.Age = age
}

type Singer struct {
	Person // extends Person by embedding it
	works  []string
}

func main() {
	var gaga = Singer{Person: Person{"Gaga", 30}}
	gaga.PrintName() // Name: Gaga
	gaga.Name = "Lady Gaga"
	(&gaga).SetAge(31)
	(&gaga).PrintName()   // Name: Lady Gaga
	fmt.Println(gaga.Age) // 31
}

From the above example, it looks that after embedding type Person, the type Singer obtains (inherits) all methods and fields of type Person, and type *Singer obtains (inherits) all methods of type *Person. Are the conclusions right? The following sections will answer this question.

Please note that, a Singer value is not a Person value, the following code doesn't compile: 
var gaga = Singer{}
var _ Person = gaga

Does The Embedding Type Obtain The Fields Of The Embedded Types?

Let's list all the fields and methods of type Singer and the methods of type *Singer used in the last example by using reflection functionalities provided by the reflect standard package: 
...
import "reflect"
...
func main() {
	t := reflect.TypeOf(Singer{}) // the Singer type
	fmt.Println(t, "has", t.NumField(), "fields:")
	for i := 0; i < t.NumField(); i++ {
		fmt.Print("   filed#", i, ": ", t.Field(i).Name, "\n")
	}
	fmt.Println(t, "has", t.NumMethod(), "methods:")
	for i := 0; i < t.NumMethod(); i++ {
		fmt.Print("   method#", i, ": ", t.Method(i).Name, "\n")
	}
	
	pt := reflect.TypeOf(&Singer{}) // the *Singer type
	fmt.Println(t, "has", pt.NumMethod(), "methods:")
	for i := 0; i < pt.NumMethod(); i++ {
		fmt.Print("   method#", i, ": ", pt.Method(i).Name, "\n")
	}
}
The result: 
main.Singer has 2 fields:
   filed#0: Person
   filed#1: works
main.Singer has 1 methods:
   method#0: PrintName
main.Singer has 2 methods:
   method#0: PrintName
   method#1: SetAge

From the result, we know that the type Singer really owns a PrintName method, and the type *Singer really owns two methods, PrintName and SetAge. But the type Singerdoesn't own a Name field. Then why is the selector expression gaga.Name legal for a Singer value gaga? Please read the next section to get the reason.

Shorthands Of Selectors

From the articles structs in Go and methods in Go, we have learned that, for a value x, x.y is called a selector, where y is either a field or a method. If y is a field, then x must be a stuct value or a struct pointer value. A selector is an expresssion, which represents a value. If the selector x.y denotes a field, it may also has its own fields (if x.y is a struct value) and methods. Such as x.y.z, where z can also be either a field or a method.

There is a syntax sugar in Go, if a middle token in a seletor corresponds an anonymous field, then that token can be omitted from the selector, except that the selector collides with (or is shadowed by) other selectors (please read the following sections for what are selector colliding and shadowing).

For example: 
package main

type A struct {
	x int
}
func (a A) MethodA() {}

type B struct {
	A
}
type C struct {
	B
}
	
func main() {
	var c C
	
	// The following 4 lines are equivalent.
	_ = c.B.A.x
	_ = c.B.x
	_ = c.A.x
	_ = c.x
	
	// The following 4 lines are equivalent.
	c.B.A.MethodA()
	c.B.MethodA()
	c.A.MethodA()
	c.MethodA()
}

This is why the expression gaga.Name is legal in the example in the last section. For it is just the shorthand of gaga.Person.Name.

As any embedding type must be a struct type, the article structs in Go has mentioned that the field of an addressable struct value can be accessed through the pointers of the struct value. So the following code is also legal in Go. 
func main() {
	var c C
	pc = &c
	
	// The following 4 lines are equivalent.
	fmt.Println(pc.B.A.x)
	fmt.Println(pc.B.x)
	fmt.Println(pc.A.x)
	fmt.Println(pc.x)
	
	// The following 4 lines are equivalent.
	pc.B.A.MethodA()
	pc.B.MethodA()
	pc.A.MethodA()
	pc.MethodA()
}

Similiarly, the selector gaga.PrintName can be viewed as a shorthand of gaga.Person.PrintName. But, it is also okay if we think it is not a shorthand. After all, the type Singer really has a PrintName method. For the similar reason, the selector (&gaga).PrintName and (&gaga).SetAge can also be viewed as, or not as, shorthands of (&gaga.Person).PrintName and (&gaga.Person).SetAge.

Note, we can also use the selector gaga.SetAge, but only if gaga is an addressable value of type Singer. It is jsut a sugar of (&gaga).SetAge. Please read method calls for details.

Implicit Methods For Embedding Types

As above has mentioned, both of type Singer and type *Singer have a PrintName method each. And the type *Singer also has a SetAge method. However, we never explicitly declare these methods for the two types. Where do these methods come from?

In fact,
  • for each method of an embedded type, if it neither collides nor is shadowed by other methods and fields (please read the following sections for details), then compilers will implicitly declare a correspoding method for the embedding struct type, and consequently, compiler will also implicitly declare a correspoding method for the unnamed pointer type whose base type is the embedding struct type.
  • and assume a struct type S embeds a named type T, for each method of type *T, if it neither collides nor is shadowed by other methods and fields, then compilers will implicitly declare a correspoding method for type *S.
Simply speaking,
  • both type struct{T} and *struct{T} will obtain (inherit) all the methods of type T.
  • type *struct{T} will obtain (inherit) all the methods of type *T as well.
  • both type struct{*T} and *struct{*T} will obtain (inherit) all the methods of type *T. As the method set of type T is a subset of type *T, we can also say both type struct{*T} and *struct{*T} will obtain (inherit) all the methods of type T.
Here are the implicitly declared methods for type Singer and *Singer and their corresponding implicitly declared functions: 
func (s Singer) PrintName() {
	s.Person.PrintName()
}

func (s *Singer) PrintName() {
	(*s).Person.PrintName()
}

func (s *Singer) SetAge(age int) {
	(&s.Person).SetAge(age) // <=> (&((*s).Person)).SetAge(age)
}

From the article methods in Go, we know that we can't explicitly declare methods for unnamed struct types and unnamed pointer types whose base types are unnamed struct types. But through type embedding, such types can also own methods.

For a type embedding, if the embedded type implements some interface types, then the embedding type will also implement these inrerface types if the embedding types obtains all the methods specified in the interface types from the embedding type.

Here is another example to show which implicit methods are declared. 
package main

import "fmt"
import "reflect"

type F func(int) bool
func (f F) Validate(n int) bool {
	return f(n)
}
func (f *F) Modify(f2 F) {
	*f = f2
}

type B bool
func (b B) IsTrue() bool {
	return bool(b)
}
func (pb *B) Invert() {
	*pb = !*pb
}

type I interface {
	Load()
	Save()
}

func main() {
	PrintTypeMethods := func(t reflect.Type) {
		fmt.Println(t, "has", t.NumMethod(), "methods:")
		for i := 0; i < t.NumMethod(); i++ {
			fmt.Print("   method#", i, ": ", t.Method(i).Name, "\n")
		}
	}
	
	var s struct {
		F
		*B
		I
	}
	
	PrintTypeMethods(reflect.TypeOf(s))
	fmt.Println()
	PrintTypeMethods(reflect.TypeOf(&s))
}
The result: 
struct { main.F; *main.B; main.I } has 5 methods:
   method#0: Invert
   method#1: IsTrue
   method#2: Load
   method#3: Save
   method#4: Validate

*struct { main.F; *main.B; main.I } has 6 methods:
   method#0: Invert
   method#1: IsTrue
   method#2: Load
   method#3: Modify
   method#4: Save
   method#5: Validate

A Defined Type Will Only Obtain Methods Which Are Obtained Through Embedding From The Source Type

In a type definition, if the new defined type is neither an interface type nor a pointer type, then for each method, which is obtained through type embedding, of the source type, a corresponding method with the same name will be implictly declared by compiler for the new defined type. The methods which are not obtained (by the source type) through type embedding will not be obtained by the new defined type.

Example: 
package main

import "fmt"

type MyInt int
func (mi MyInt) IsOdd() bool {
	return mi%2 == 1
}

// Type S has two methods, IsOdd and Scale.
// IsOdd is obtained through embedding MyInt.
type S struct {
	MyInt
}
func (s S) Scale(n int) int {
	return int(s.MyInt) * n
}

// Type T only obtains the IsOdd method from S.
// It doesn't obtain the Scale method from S.
type T S

func main() {
	var t T = T{5}	
	fmt.Println(t.IsOdd())   // okay, print true.
	/*
	fmt.Println(t.Scale(12)) // error: t.Scale undefined
	*/
}

Selector Shadowing And Colliding

The number of dots in the full form of a selector can be called the depth of the selector. For example, if x.y.z is a full form selector, then its depth is 2.

Sometimes, for a value x, there may be multiple full form selectors whose last tokens are identical. Assume the last tokens are y, among these selectors,
  • if only one selector has the smallest depth, then it can be shortened to x.y (or one-dot selector), whereas the others can't. We say the others are shadowed by the one with the smallest depth.
  • if more than one selectors have the smallest depth, then none of them can be shortened to x.y. We say the selectors with the smallest depth are collided.

If a method selector is shadowed by another method selector, and the two corresponding method signatures are same, we say the first method is overridden by the other one.

If a method selector with depth larger than one can't be shortened to an one-dot selector, then there will not be corresponding implicit methods declared for it.

For example, assume A, B and C are three defined types. 
type A struct {
	x string
}
func (A) y(int) bool {
	return false
}

type B struct {
	y bool
}
func (B) x(string) {}

type C struct {
	B
}
The following code doesn't compile. The reasons is selector v1.A.x and v1.B.x are collided, so both of them can't be shortened to v1.x. The same sotuation is for selector v1.A.y and v1.B.y. There is no implicit methods are declared for the type of v1. 
var v1 struct {
	A
	B
}

func f1() {
	_ = v1.x
	_ = v1.y
}
The following code compiles okay. The selector v2.C.B.x is shadowed by v2.A.x, so the selector v2.x is a shorten form of v2.A.x actually. For the same reason, the selector v2.y is a shorten form of v2.A.y, instead of v2.C.B.y. 
var v2 struct {
	A
	C
}

func f2() {
	fmt.Printf("%T \n", v2.x) // string
	fmt.Printf("%T \n", v2.y) // func(int) bool
}

Interface Types Embed Interface Types

Not only can struct types embed other types, but also interface types can. But interface types can only embed named interface types. Please read interfaces in Go for details.

A Detail About Selector Shorthands

A pointer value of a defined one-level struct pointer type can access the fields of the struct value it references, But two-level (and higher level) pointers, either defined or not, can't. 

package main

type T struct {
	x int
}
func (T) m(){} // T has one method.

type P *T  // a defined one-level pointer type.
type PP *P // a defined two-level pointer type.

func main() {
	var t T
	var p P = &t
	var pp PP = &p
	p.x = 123  // okay
	pp.x = 789 // error: type PP has no field or method x
}
